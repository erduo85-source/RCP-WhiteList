## 2. 页面定位

### 2.1 页面角色

“多维画像查询”入口页是一个**查询前轻量工作台首页**，用于帮助产品、运营等非技术角色：

1. 快速发起按账号、设备、IP 的画像查询
2. 快速从近期高风险对象中进入核实流程

它不是查询后结果页，也不是风险分析总览页，更不是策略配置页。

### 2.2 与查询后结果页的职责边界

| 页面 | 主要职责 |
| --- | --- |
| 查询前入口页 | 发起查询、回看历史查询、进入待核实对象 |
| 查询后结果页 | 展示场景总览、场景核实、核实依据、时间线、日志、关联分析 |

### 2.3 设计原则

| 原则 | 说明 |
| --- | --- |
| 轻量优先 | 首页只保留发起查询和进入对象所需的最小信息 |
| 不抢职责 | 不在首页提前展开场景分析、风险解释、时间线、日志等重内容 |
| 可直接进入 | 首页列表中的每个对象都应可快速进入查询后结果页 |
| 稳定清晰 | 页面结构固定，避免因对象类型或状态不同造成明显跳变 |

## 3. 整体结构

页面采用瀑布流三段结构：

| 顺序 | 区域 | 作用 |
| --- | --- | --- |
| 1 | 页眉 | 说明页面名称与能力范围 |
| 2 | 搜索区 | 发起查询，并提供历史查询快捷入口 |
| 3 | 列表区 | 展示近期高风险对象，作为轻量入口列表 |

### 3.1 结构说明

1. 页眉只承担识别和说明功能，不放复杂操作。
2. 搜索区是首页主视觉中心，优先于列表区。
3. 列表区作为“待核实对象入口”，信息密度应明显低于查询后结果页。

## 4. 页眉方案

| 项目 | 方案 |
| --- | --- |
| 标题 | `多维画像查询` |
| 说明文案 | 支持从账号、设备、IP 维度进行风险分析，统一查看画像、交易、行为等风险数据 |
| 操作 | 首版不额外增加页眉右侧功能按钮 |

### 4.1 页眉目标

1. 让用户快速理解页面用途
2. 不引入额外认知负担
3. 为搜索区留出足够视觉注意力

## 5. 搜索区方案

### 5.1 查询维度切换

搜索区采用三页签切换：

| 页签 | 说明 |
| --- | --- |
| 按账号查询 | 适合通过 SDKID 或账号名发起查询 |
| 按设备查询 | 适合通过设备ID发起查询 |
| 按 IP 查询 | 适合通过 IP 地址发起查询 |

### 5.2 账号查询表单

账号查询采用紧凑布局：

`标识类型下拉 + 输入框 + 查询按钮`

| 项目 | 方案 |
| --- | --- |
| 标识类型下拉 | `SDKID`、`账号名` |
| 输入框占位文案 | 根据下拉项切换，如“请输入SDKID”或“请输入账号名” |
| 查询按钮 | 主按钮，文案统一为 `查询` |

说明：

1. 账号查询不拆成多个独立输入框，避免首屏过重。
2. 用“下拉 + 单输入框”兼顾紧凑性和明确性。

### 5.3 设备查询表单

| 项目 | 方案 |
| --- | --- |
| 输入项 | `设备ID` |
| 占位文案 | `请输入设备ID` |
| 操作按钮 | `查询` |

### 5.4 IP查询表单

| 项目 | 方案 |
| --- | --- |
| 输入项 | `IP地址` |
| 占位文案 | `请输入IP地址` |
| 操作按钮 | `查询` |

### 5.5 搜索提交口径

| 项目 | 定稿 |
| --- | --- |
| 提交后行为 | 直接打开查询结果大弹窗 |
| 是否进入中间页态 | 否 |
| 首版输入校验 | 仅校验非空 |
| 首版是否做格式强校验 | 否 |

### 5.6 搜索提示文案

| 场景 | 文案 |
| --- | --- |
| 空值提交 | `请输入查询内容` |
| 查无结果 | `未查询到相关对象` |
| 查询失败 | `查询失败，请稍后重试` |

说明：

1. 查无结果和失败都用首页轻提示承接。
2. 首页不展开错误面板，也不强行打开空结果弹窗。

## 6. 历史查询方案

历史查询位于搜索区下方，作为辅助入口，不抢搜索主流程。

| 项目 | 方案 |
| --- | --- |
| 位置 | 搜索区底部 |
| 表现形式 | 标签流 / 胶囊标签 |
| 内容来源 | 首版直接使用 mock 数据 |
| 支持对象 | 账号、设备、IP 混合展示 |
| 保留条数 | 最多保留 8 条 |
| 操作 | 点击后自动回填并立即查询 |
| 辅助操作 | 右侧提供“清空记录” |
| 去重逻辑 | 首版不实现 |

### 6.1 历史查询的作用

1. 让高频核实对象可被快速再次查询
2. 减少重复输入
3. 作为首页的轻辅助信息，而不是主列表内容

### 6.2 历史查询边界

| 项目 | 约束 |
| --- | --- |
| 是否承担分析职责 | 不承担 |
| 是否展示风险分 | 不展示 |
| 是否展示复杂筛选 | 不展示 |
| 是否实现真实写入与持久化 | 首版不要求 |

## 7. 列表区方案

### 7.1 区块定位

列表区标题定为：

`近期高风险列表`

该列表不是分析表，而是**轻量待核实对象列表**。

### 7.2 列表职责

| 职责 | 说明 |
| --- | --- |
| 告诉用户现在有哪些对象值得优先关注 | 用轻量摘要呈现 |
| 支持用户快速进入对象详情核实 | 通过单一动作完成 |
| 不在首页展开复杂解释 | 避免与查询后结果页重叠 |

### 7.3 对象组织方式

列表采用**混合对象列表**：

| 对象类型 | 是否纳入 |
| --- | --- |
| 账号 | 是 |
| 设备 | 是 |
| IP | 是 |

#### 7.3.1 区分方式

通过对象图标区分不同类型：

| 类型 | 区分方式 |
| --- | --- |
| 账号 | 账号图标 |
| 设备 | 设备图标 |
| IP | 地球 / 网络图标 |

说明：

1. 首版不按对象类型再分 Tab。
2. 同一表内混排，降低首页结构复杂度。
3. 图标只承担“类型识别”，不承载额外状态语义。

### 7.4 列表字段定稿

首版列表固定为 6 列：

| 列名 | 说明 |
| --- | --- |
| 对象 | 对象主标识，配合图标区分类型 |
| 近7日风险事件数 | 风险强度摘要 |
| 近7日最高风险分 | 风险上限摘要 |
| 近7日平均风险分 | 风险水平辅助摘要 |
| 最近风险时间 | 最近一次风险事件发生时间 |
| 操作 | 进入查询后结果页 |

#### 7.4.1 字段说明

| 字段 | 设计目的 |
| --- | --- |
| 对象 | 告诉用户当前行是谁 |
| 近7日风险事件数 | 帮助快速判断近期风险密度 |
| 近7日最高风险分 | 帮助快速判断风险峰值 |
| 近7日平均风险分 | 帮助粗看整体风险水平 |
| 最近风险时间 | 帮助判断风险是否接近当前 |
| 操作 | 明确进入详情的唯一动作 |

#### 7.4.2 对象列显示口径

| 项目 | 定稿 |
| --- | --- |
| 显示方式 | 只显示主标识 |
| 账号 | 显示账号名或 SDKID 之一 |
| 设备 | 显示设备ID |
| IP | 显示IP地址 |

#### 7.4.3 当前不纳入的字段

| 字段 | 不纳入原因 |
| --- | --- |
| 对象归属信息 / 所属信息 | 会显著增加首页密度 |
| 风险等级标签 | 容易把首页再次拉向分析页 |
| 最近操作时间 | 对首页进入决策价值有限 |
| 场景分布 | 属于查询后结果页范畴 |
| 推荐查看场景 | 属于查询后结果页范畴 |

### 7.5 列表交互方案

#### 7.5.1 行级交互

| 项目 | 方案 |
| --- | --- |
| 整行点击 | 不支持 |
| 原因 | 避免误触，保持首页交互克制 |
| 唯一主动作 | 右侧按钮 |

#### 7.5.2 按钮方案

| 项目 | 方案 |
| --- | --- |
| 按钮文案 | `立即查询` |
| 按钮位置 | 每行最右侧 |
| 按钮动作 | 进入该对象的查询后结果页 / 结果弹窗 |

说明：

1. 首页不做多动作并列。
2. “立即查询”比“详情”更符合从首页进入核实流程的语义。
3. 按钮是列表的唯一强操作锚点。

### 7.6 分页与排序口径

| 项目 | 定稿 |
| --- | --- |
| 分页 | 保留分页展示 |
| 每页条数 | 固定 10 条 / 页 |
| 是否支持切换页容量 | 否 |
| 默认排序 | 按近7日风险事件数倒序 |

说明：

1. 首页列表虽然轻量，但仍保留可浏览能力。
2. 默认按风险事件密度排序，优先展示近期更值得核实的对象。

## 8. 空态与状态方案

### 8.1 默认态

默认态指首页正常加载完成，存在历史查询和/或列表数据时的状态。

| 区域 | 表现 |
| --- | --- |
| 搜索区 | 展示页签和默认查询表单 |
| 历史查询 | 若有记录则展示 |
| 列表区 | 展示近期高风险列表 |

### 8.2 列表空态

当“近期高风险列表”为空时，采用引导式空态：

`暂无高风险对象，建议通过上方条件发起查询`

#### 8.2.1 空态原则

| 原则 | 说明 |
| --- | --- |
| 不做报错语义 | 不是异常，不应显得页面坏了 |
| 不做结果结论 | 不表示“系统正常”或“没有问题” |
| 继续引导动作 | 明确告诉用户可以从上方继续发起查询 |

### 8.3 历史查询空态

| 场景 | 方案 |
| --- | --- |
| 无历史查询 | 直接不展示记录标签流，或展示极轻占位提示 |

### 8.4 加载态与异常态

首页首版仅需保留基础状态位：

| 状态 | 方案 |
| --- | --- |
| 列表加载中 | 表格骨架屏 / 行骨架 |
| 列表加载失败 | 轻量失败提示 + 重试入口 |
| 搜索提交中 | 查询按钮 loading |

说明：

1. 加载态要轻，不要比查询后结果页更重。
2. 异常态只处理“首页数据拉取失败”，不承载风险解释。
3. 搜索查无结果或失败，统一使用 message/toast 轻提示。

## 9. 与查询后结果页的边界约束

首页明确不纳入以下内容：

| 模块 | 原因 |
| --- | --- |
| 场景总览条 | 属于查询后多场景核实导航 |
| 场景 Tab | 属于查询后核实流程 |
| 当前场景概览卡 | 属于查询后分析结论层 |
| 核实依据区 | 属于查询后解释层 |
| 时间线 | 属于查询后事件核查层 |
| 日志区 | 属于查询后明细核查层 |
| 关联分析区 | 属于查询后深挖分析层 |

## 11. 低保真结构说明

从上到下结构示意如下：

1. 页眉
2. 搜索区页签
3. 当前查询表单
4. 历史查询记录
5. 近期高风险列表
6. 列表右侧“立即查询”动作

### 11.1 结构关系

| 顺序 | 模块 | 关系说明 |
| --- | --- | --- |
| 1 | 页眉 | 只做识别，不参与交互主链路 |
| 2 | 搜索区 | 首页主操作区 |
| 3 | 历史查询 | 搜索区辅助入口 |
| 4 | 列表区 | 搜索之外的次操作入口 |

## 12. 实现建议

若进入开发阶段，建议入口页单独作为一个页面模块处理，与查询后结果页分离。

### 12.1 建议文件方向

| 类型 | 文件建议 | 作用 |

| 类型 | `src/features/profile-query/entry/types.ts` | 首页数据结构定义 |
| 样式 | `src/features/profile-query/entry/style.css` 或现有样式拆分 | 首页专属样式 |

### 12.2 数据结构建议

| 类型 | 关键字段 |
| --- | --- |
| 查询状态 | 当前页签、账号标识类型、输入值 |
| 历史查询项 | 对象类型、对象标识 |

## 13. 验收标准

| 维度 | 验收标准 |
| --- | --- |
| 页面定位 | 用户能理解这是查询前首页，而不是结果页 |
| 结构 | 页眉、搜索区、列表区层级清楚 |
| 搜索 | 三类查询可切换，账号查询支持 `SDKID / 账号名` |
| 搜索提交 | 点击查询后直接进入结果大弹窗 |
| 历史查询 | 可点击回填并自动再次查询 |
| 列表 | 账号、设备、IP 可混排，图标区分清晰 |
| 交互 | 列表不支持整行点击，只能通过 `立即查询` 进入 |
| 分页 | 列表固定 10 条 / 页，支持分页浏览 |
| 空态 | 空态文案具备引导性，不带报错或结论语义 |
| 边界 | 首页不出现场景总览、时间线、日志、关联分析等结果页模块 |

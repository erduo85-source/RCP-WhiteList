## 问询记录

| 轮次 | 主题 | 关键问题 | 用户结论 | 对方案的影响 |
| --- | --- | --- | --- | --- |
| 1 | 页面定位 | 首版“多维画像查询”服务谁、解决什么动作？ | 首版偏向产品/运营使用。产品包括产品负责人和产品领导，要求页面高可用、符合产品规范；运营主要用于游戏项目客诉场景，如盗号反馈后的后台快速核实。 | 页面定位不是策略配置后台，而是高可信、高可读、适合非技术角色快速判断的核实页面。 |
| 2 | 查询对象 | 首版主入口偏单用户还是偏人群？ | 选择单个查询对象为主。 | 页面首屏应优先服务单对象核实，而不是复杂分群筛选。 |
| 3 | 查询入口 | 最常用哪个字段发起查询？ | 采用统一查询入口 + 类型选择，支持账号信息（账号名、SDKID）、设备信息（设备ID）、网络信息（IP地址）。 | 页面头部应采用统一检索框和查询类型切换，而不是分散入口。 |
| 4 | 结果范围 | 查询成功后首版需要展示哪些信息？ | 需要包含基础信息、关联信息、日志信息，以及待补充内容；其中关联信息要支持图谱交互与下钻，日志要能快速定位风险登录/注册/支付触发。 | 结果页不能只有基础详情，需要兼顾关系洞察和日志定位，但需控制优先级。 |
| 5 | 第一屏任务 | 第一屏最优先回答什么问题？ | 第一屏最优先回答“最近是否发生了异常登录、异常设备、异常 IP 变化”。进入页面时通常已带着问题而来，主要用于盗号核实和误封核实。身份核实由客诉人工先完成，后台重点查看风险信息。 | 页面第一屏必须围绕“近期异常变化核实”组织，而不是先堆基础资料。异常时间线、异常设备/IP、风险触发应高优先级前置。 |
| 6 | 首屏主模块 | 如果第一屏只能有一个最占视觉比重的主模块，应是什么？ | 选择异常概览卡片区。希望先快速告诉用户：近期登录次数、其中安全/风险登录次数、设备使用情况、常用设备与高风险设备、IP 使用情况、常用 IP 与高风险 IP。用户同时指出，目前这套表述仍然偏字段罗列，不够精炼，不够像“概览”。 | 首屏需要先做“摘要判断”，不能直接把登录/设备/IP 指标平铺出来。需要进一步抽象为少量可快速扫读的核实结论。 |
| 7 | 概览表达方式 | 首屏概览更偏哪种呈现风格？ | 选择混合式：左侧先给结论，右侧给少量关键指标。 | 首屏概览应同时具备“先告诉用户怎么看”和“立即给到支撑数字”两种能力。 |
| 8 | 样例参考 | 过往私聊回复样例能提供什么页面设计参考？ | 用户提供了盗号核实私聊截图，强调该内容不能直接用于页面展示，只能抽象成设计参考。样例中高频出现的结构包括：账号历史特征、被盗登录时间线、登录情况补充、风控模型判断。 | 页面设计可以借鉴“先概括，再给证据，再按时间线展开”的结构，但必须从私聊叙述抽象为标准化后台模块，不能直接照搬话术。 |
| 9 | 结论边界 | 首屏是否直接给“更像盗号 / 更像误封 / 暂不明确”的倾向判断？ | 当前选择仅给“是否存在异常”。原因是后台未来将面向多个游戏项目组，异常类型很多，首期主要支持防盗号，后续还会有防刷号等需求；现阶段只有盗号策略算法，无法可靠覆盖误封、账号买卖纠纷等场景。 | 页面首屏不能假装全知。需要把“异常发现”和“异常定性”分层，并明确展示当前判断能力的边界。 |
| 10 | 算法现状 | 现有自动化能力的边界是什么？ | 当前已具备盗号风险打分能力，相关依据来自本地文档 `doc/风控平台V1.3.1-需求文档.md` 中的登录地异常、登录地变化过快、登录设备异常等规则；尚不具备可泛化到所有异常类型的统一倾向判断能力。 | 页面可复用“风险打分 + 风险证据”的表达方式，但不应在首版承诺跨异常类型的自动结论。 |
| 11 | 策略模型现状 | 项目下的风控策略如何定义风险与处置？ | 每个项目支持配置一个风控策略/判断模型；模型本身不会直接输出明确的低中高风险，而是根据得分区间匹配不同处置动作。 | 页面不能简单把“模型分”直接等同于“统一风险等级”，需要区分“事件得分”“区间命中”“处置结果”。 |
| 12 | 账号视角聚合 | 一个账号近期有大量登录且每次得分不同，首屏如何展示？ | 用户提出典型问题：例如近期 101 次登录，分值 0~100 不等，不能直接确定是取最高分、平均分还是其他映射方式。 | 首屏需要明确“单次事件分”到“账号近期风险概览”的聚合口径，否则结论会误导。 |
| 13 | 时间窗口 | 首屏概览默认看哪个时间范围？ | 选择双时间窗：近 7 天异常 + 近 30 天历史基线。 | 页面应同时表达“最近是否出事”和“历史常态是什么”，后续概览、时间线、基线对比都应围绕这两个时间窗组织。 |
| 14 | 首屏指标主轴 | 首屏右侧关键指标优先突出哪一组？ | 选择混合主轴：登录异常 + 设备异常 + IP 异常，各放 1 到 2 个核心指标。 | 首屏右侧不应被单一风险分或全量指标占据，而应服务核实场景，提供跨维度的少量高价值支撑数字。 |
| 15 | 多场景组织 | 登录、注册、支付三类风险是否统一概览？ | 用户认可“先做总览摘要，下面再接场景 Tab”的方向，但进一步质疑：如果 Tab 已能体现异常集中场景，总览摘要是否还有必要保留，或应替换成其他更合适的信息。 | 页面首屏不应简单叠加“总览摘要 + Tab”而造成重复，需要重新定义总览区的独特价值。 |
| 16 | 场景默认与排序 | 如果根据异常命中自动默认最相关场景，多个异常时如何排序？ | 用户认可“根据当前查询对象和异常命中情况自动默认最相关场景”的方向，但要求明确多异常并存时的排序逻辑。 | 页面需要定义场景优先级规则，避免默认场景和 Tab 顺序不可解释。 |
| 17 | 总览区形态 | 总览摘要是否保留，应该替换成什么？ | 选择将“总览摘要”升级为“场景总览条”。并要求将排序逻辑单独沉淀到新文档 `风控平台V1.3.3-需求文档.md`。 | 首屏顶部不再只是文本摘要，而是承担“异常总况 + 场景分布 + 首看建议 + 排序依据”的导航职责。 |
| 18 | Tab 呈现方式 | 场景 Tab 是动态排序还是固定顺序？ | 选择固定顺序，但在推荐场景 Tab 上增加“推荐查看”标记。 | 页面结构保持稳定，降低认知跳变；同时通过推荐标记引导用户优先查看最相关场景。 |
| 19 | 场景总览条密度 | 场景总览条最少展示哪些字段？ | 选择平衡版：核实状态 + 各场景异常数 + 推荐场景 + 推荐依据一句话。 | 场景总览条只保留导航决策信息，不承载事件、设备、IP 等证据细节，避免与下方场景内容重复。 |
| 20 | 推荐依据文案 | 场景总览条中的推荐依据一句话采用什么风格？ | 选择混合型：既保留规则解释，也保留业务导向表达。 | 推荐依据文案应兼顾可解释性和可读性，避免过于算法化或过于空泛。 |
| 21 | 登录异常首模块 | 进入“登录异常” Tab 后，场景总览条下方的第一个主模块是什么？ | 选择“登录异常概览卡”。同时补充：需要明确无异常时该模块如何显示。 | 登录异常 Tab 首屏仍应先给场景级概览，而不是直接落到时间线或全量日志；同时必须补齐空异常/无明显异常状态设计。 |
| 22 | 登录异常无异常表达 | 登录异常 Tab 在“近7天无明显异常”时，概览卡采用什么表达？ | 选择混合型：`近7天未发现明显登录异常，当前证据不足以支持盗号判断`。 | 无异常状态不是空白结果，而是有判断边界的核实结论，需要既避免误导“绝对没问题”，又避免过度保守。 |
| 23 | 登录异常第二模块 | 登录异常概览卡之后，先看哪块？ | 选择“异常时间线”。 | 登录异常场景下，用户在看完概览后，优先需要理解“异常是如何发生的”，再去查看支撑异常判断的设备/IP 变化与风险明细。 |
| 24 | 场景差异化结构 | 登录异常适合在时间线后看设备/IP 变化，但注册刷号等场景是否也沿用同一模块？ | 用户指出：盗号场景下设备/IP 变化卡合理，但注册刷号场景没有“常用设备”概念，不能直接套用相同模块。 | 页面不应为所有场景强行复用同一第三模块，应改为“场景专属证据模块”，由不同场景各自定义证据结构。 |
| 25 | 证据模块命名 | 第三模块的统一产品名叫什么？ | 选择“核实依据卡”。 | 该命名可同时覆盖“有异常时的判断依据”和“无明显异常时的核实依据”，比“异常证据卡”更中性、更适合多场景。 |
| 26 | 登录异常第四模块 | “核实依据卡”之后，更希望先看哪块？ | 选择“全量登录日志”，但强调不能只看异常，否则无法判断何为异常；同时也不能让普通数据淹没异常数据。建议支持默认筛选为风险登录，并允许手动切换到全量。 | 第四模块应保留完整历史上下文，但默认交互要优先帮助用户定位风险数据，不能直接把全量原始日志平铺给用户。 |
| 27 | 异常时间线交互 | 既然已经有“异常时间线”，它的交互应该是什么样，是否有助于快速定位风险数据？ | 用户主动提出该问题，要求评估时间线与日志区的关系，避免功能重复；并进一步选择时间线节点不与日志区直接联动，原因是中间隔着“核实依据卡”，垂直滚动下视觉不连贯。 | 需要明确“异常时间线”的独立价值：它应服务快速理解异常经过，本身可承载节点详情，但不依赖跨模块跳转日志。 |
| 28 | 时间线节点详情 | 异常时间线点击节点后，详情采用什么呈现方式？ | 选择节点下方内联展开。 | 时间线应在当前位置完成关键事件浏览，避免二次弹层或抽屉打断阅读路径。 |
| 29 | 查询结果容器形态 | 查询结果是普通页面还是可复用独立组件？ | 用户补充希望把查询结果做成可在其他页面快速调用并展示的独立弹窗组件，因此不适合弹窗盖弹窗、弹窗盖抽屉。 | 结果容器需要按“独立弹窗组件”设计，内部交互应避免再次叠加重型浮层，优先使用页内展开、折叠、切换等方式。 |
| 30 | 查询结果弹窗形态 | 独立查询结果组件采用哪种容器形态？ | 初步考虑全屏弹窗，后修正为大弹窗。原因是当前后台没有全屏组件规范，引入新组件得不偿失。 | 查询结果组件应优先复用现有后台弹窗规范，以较低成本落地；页面结构和信息密度需按大弹窗空间重新约束。 |
| 31 | 弹窗标题区 | 大弹窗顶部固定区需要保留哪些信息？ | 用户补充：弹窗已有固定标题区域，包含名称和关闭按钮；名称可以直接带查询对象。 | 结果弹窗顶部不必重复设计完整对象头部，应优先复用现有标题规范，把查询对象融入标题文本。 |
| 32 | 弹窗上半区顺序 | 标题区下面，应先放场景总览条还是先放场景 Tab？ | 选择先放场景总览条，再放场景 Tab。 | 结果组件应先给整体核实导航，再进入具体场景，符合“先判断全局，再下钻场景”的认知顺序。 |
| 33 | 基础信息区默认状态 | 基础信息/补充信息区默认如何展示？ | 选择“只展示极少基础字段，其余折叠”。 | 大弹窗中前半部分已承载主要核实链路，基础信息应尽量轻量前置，避免挤占首屏与主线内容。 |
| 34 | 基础信息区位置 | 基础信息应放在什么位置？ | 用户补充：基础信息区域改为默认收起的一行展示，并置顶保留。 | 基础信息不应沉到底部，而应作为轻量背景信息固定在上方，方便随时查看且不抢占主视区。 |
| 35 | 关联信息价值 | 关联信息在新方案中是否仍应保留？ | 用户补充：关联信息有助于进一步发现问题，例如一个设备登录了 100 个账号，存在工作室嫌疑。 | 关联信息不应被完全降级为可有可无的图谱展示，而应作为深挖异常的重要分析模块保留。 |
| 36 | 登录异常时间线节点规则 | 登录异常时间线应如何定义关键节点？ | 用户结论：只对风险节点做特殊标记即可。 | 时间线不再预设过多节点类型，避免复杂化；首版重点是帮助用户快速识别风险事件。 |
| 37 | 登录异常概览卡字段 | 登录异常场景概览卡首期保留哪些字段？ | 用户最终结论：改为纯数值指标卡，保留“近7天风险登录次数”“近7天最高风险分”“近7日平均风险分”。并明确指出：场景总览条已经表达哪些场景有风险，进入具体场景后没必要再重复布尔型“是否存在异常”字段；新设备/IP也不等同于风险；其中“近7日平均风险分”仅基于风险登录事件计算，不纳入全量登录事件。 | 概览卡从“结论 + 指标”进一步收敛为“纯数值指标卡”，布尔判断和解释性信息分别留在场景总览条与核实依据区。 |
| 38 | 关联图节点悬浮字段 | 关联图节点移入时展示哪些信息？ | 用户结论：展示“对象唯一标识 + 近7天风险登录次数 + 近7天最高风险分 + 近7日平均风险分”。 | 图谱节点悬浮信息与登录异常概览卡口径保持一致，既方便快速判断，也避免不同模块口径不统一。 |
| 39 | 场景总览条文案 | 场景总览条应如何写示例文案？ | 已沉淀为四段式表达：核实状态、场景分布、推荐场景、推荐依据；并补充了登录异常主导、多场景并存、无明显异常、注册异常主导等示例文案。 | 场景总览条的结构和语气已可直接用于设计稿与开发，不再停留在抽象字段层。 |

## 当前已收敛结论

| 主题 | 当前结论 |
| --- | --- |
| 页面核心角色 | 产品负责人 / 产品领导 / 游戏运营 |
| 页面核心用途 | 面向带问题进入的单对象核实，重点支持盗号核实与误封核实 |
| 页面入口形态 | 统一查询入口 + 查询类型选择 |
| 首屏主任务 | 判断近期是否存在异常登录、异常设备、异常 IP 变化 |
| 页面调性要求 | 高可用、可快速判断、符合后台规范、适合评审 |
| 首屏主模块 | 异常概览卡片区 |
| 概览表达方式 | 混合式：左侧结论，右侧关键指标 |
| 样例参考价值 | 可抽象出“历史基线 / 异常时间线 / 补充说明 / 风险判断”四类信息结构 |
| 结论边界 | 首版只给“是否存在异常”，不直接给跨异常类型的自动定性结论 |
| 现有算法能力 | 已有盗号风险打分，可支撑风险等级和证据展示 |
| 策略模型特点 | 现有模型更接近“按分值区间触发处置”，不是直接输出统一等级 |
| 默认时间口径 | 近 7 天异常 + 近 30 天历史基线 |
| 首屏指标结构 | 混合主轴，覆盖登录 / 设备 / IP 三类异常 |

## 概览模块结论

| 主题 | 结论 |
| --- | --- |
| 模块定位 | 首屏核心模块是“异常概览”，用于帮助产品/运营在进入页面后快速形成初判。 |
| 回答问题 | 概览优先回答“近 7 天是否存在异常”以及“异常相对近 30 天历史基线是否明显偏离”。 |
| 结论方式 | 左侧给通用结论，如“存在异常 / 暂无明显异常”；右侧给少量支撑指标。 |
| 指标原则 | 不展示单一总分，不直接拿平均分或最高分代表账号整体风险。 |
| 聚合口径 | 账号视角应由“近期最高风险事件 + 风险事件分布 + 最近风险时间”组合表达。 |
| 时间窗口 | 默认同时看近 7 天异常和近 30 天历史基线。 |
| 风险维度 | 当前首屏指标已收敛为登录 / 设备 / IP 三类混合指标，但后续需评估注册 / 支付场景如何纳入。 |
| 多场景结构方向 | 倾向“场景总览 + 场景 Tab”，但总览区需避免与 Tab 重复。 |
| 总览区最终方向 | 升级为“场景总览条” |
| Tab 交互方向 | Tab 顺序固定，通过“推荐查看”标记表达优先级 |
| 场景总览条字段 | 平衡版字段集合：核实状态、各场景异常数、推荐场景、推荐依据一句话 |
| 推荐依据风格 | 混合型：业务提示 + 规则解释 |
| 登录异常首模块 | 登录异常概览卡 |
| 登录异常无异常状态 | 混合型结论：未发现明显异常，但当前证据不足以支持盗号判断 |
| 登录异常前两块顺序 | 先“登录异常概览卡”，再“异常时间线” |
| 跨场景结构原则 | 场景共享骨架，但第三层证据模块必须允许按场景差异化 |
| 第三模块统一命名 | 核实依据卡 |
| 登录异常前四块方向 | 登录异常概览卡 → 异常时间线 → 核实依据卡 → 全量登录日志（默认风险筛选，可切全量） |
| 时间线详情方式 | 节点下方内联展开 |
| 结果容器方向 | 作为可复用独立弹窗组件设计，内部避免再次叠加弹窗/抽屉 |
| 结果容器形态 | 大弹窗，遵循现有后台组件规范 |
| 时间线节点规则 | 只对风险节点做特殊标记 |
| 登录异常概览卡字段 | 近7天风险登录次数、近7天最高风险分、近7日平均风险分 |
| 标题区约束 | 复用弹窗固定标题区，标题可直接带查询对象 |
| 弹窗上半区顺序 | 标题区 → 场景总览条 → 场景 Tab |
| 基础信息区策略 | 基础信息区域本身置顶保留，默认收起，收起态为一行三列概览 |
| 关联信息定位 | 作为进一步发现问题的重要分析模块保留 |

## 新方案相对旧方案的调整点（补充）

| 调整点 | 旧方案 | 新方案 | 为什么要改 | 好处 |
| --- | --- | --- | --- | --- |
| 结果容器定位 | 更像页面内的大块详情内容 | 调整为可复用的大弹窗查询结果组件 | 需要在其他页面快速调用，且不希望为单一页面定制过重结构 | 更利于复用，调用成本低，符合现有后台规范 |
| 顶部信息组织 | 顶部先放查询对象与基础信息 | 弹窗标题区仅承载标题和关闭，标题可直接带查询对象；标题下先进入核实导航 | 旧方案对象信息占位较重，容易抢走核实主线 | 顶部更轻，用户更快进入“异常判断”流程 |
| 信息架构主线 | 基础信息 → 标签 → 关联信息 → 日志 | 场景总览条 → 场景 Tab → 场景概览卡 → 异常时间线 → 核实依据卡 → 日志 → 基础信息 | 旧方案更像静态资料页，不适合带问题核实 | 更符合运营/产品“先判断、再下钻、再核查”的路径 |
| 首屏重点 | 先展示账号信息和标签信息 | 先展示异常总况和推荐查看场景 | 首期场景是带问题进入核实，不是先看全量资料 | 首屏决策更快，减少无关信息干扰 |
| 多场景组织 | 未明确区分登录/注册/支付风险场景 | 明确采用“场景总览条 + 固定顺序场景 Tab” | 不同风险场景判断口径不同，不能混成一个概览 | 兼容盗号、防刷号、支付异常等扩展场景 |
| 场景优先级表达 | 未体现“先看哪个场景” | 固定 Tab 顺序，但通过“推荐查看”标记引导优先级 | 动态改 Tab 顺序会造成结构跳变；不提示优先级又不利于核实 | 结构稳定，可解释性强，用户知道先看哪里 |
| 概览表达方式 | 更偏字段罗列和资料陈列 | 调整为“左侧结论 + 右侧少量关键指标”的场景概览 | 字段堆叠不等于概览，无法快速形成初判 | 更快给出结论，同时保留支撑数字 |
| 结论边界 | 容易被理解为直接给最终业务定性 | 明确只先给“是否存在异常”，不假装能覆盖所有异常类型自动定性 | 现阶段只有部分场景模型，无法稳定判断误封、买卖号、代充等复杂场景 | 产品表达更诚实，后续可渐进增强 |
| 时间口径 | 未明确近因与历史基线的对照 | 明确为“近7天异常 + 近30天历史基线”双时间窗 | 只看近期会缺少常态对照，只看长期又不利于核实时效 | 同时支持“最近出事没有”和“是否偏离常态”两类判断 |
| 时间线定位 | 旧方案以日志列表为主，时间线价值不突出 | 异常时间线升级为独立主模块，负责讲清异常经过 | 日志表不适合快速理解事件演进 | 用户更容易先看懂发生了什么 |
| 时间线交互 | 旧方案未体现 | 节点下方内联展开详情，不跨模块联动日志 | 大弹窗内部再做远距离跳转会打断阅读路径 | 交互连续，阅读更顺，避免弹窗内再叠复杂交互 |
| 证据模块设计 | 关联图谱为主，偏静态可视化 | 调整为“核实依据卡”，并允许按场景差异化 | 登录、注册、支付场景的证据结构不同，图谱也不是所有场景的最高优先级 | 结构更通用，能覆盖有异常和无明显异常两种状态 |
| 图谱优先级 | 关联图谱位置靠前 | 图谱不再作为首屏核心主线，可后置为补充能力 | 图谱解释成本高，不一定有助于首轮快速核实 | 首屏更聚焦，后续仍可保留图谱作为深挖工具 |
| 日志查看方式 | 日志直接作为主承载区之一 | 保留全量日志，但默认查看风险记录，可手动切全量，并持续高亮风险 | 只看异常会失去上下文，只看全量又会淹没风险 | 兼顾效率和完整性，更适合核查 |
| 基础信息区权重 | 基础信息和标签信息前置且占位较大 | 调整为仅保留少量前置基础字段，其余折叠或后置 | 基础信息不是首轮核实的核心判断依据 | 减少首屏噪音，把空间让给异常主线 |

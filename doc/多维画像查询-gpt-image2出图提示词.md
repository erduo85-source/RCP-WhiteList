# 多维画像查询 gpt image2 出图提示词

## 1. 文档目的

| 项目 | 内容 |
| --- | --- |
| 目标 | 为 `gpt image2` 提供稳定、可复用的出图提示词 |
| 用途 | 基于已确认的低保真结构稿，生成首轮高保真设计图 |
| 原则 | 先锁结构，再收视觉；不在出图阶段反复改产品逻辑 |

## 2. 出图前提

以下内容已经确定，出图时不再改动：

| 项目 | 已确认内容 |
| --- | --- |
| 容器形态 | 大弹窗 |
| 顶部顺序 | 标题区 → 基础信息区域（默认收起）→ 场景总览条 → 场景 Tab |
| 主线结构 | 场景概览卡 → 核实依据区 → 时间线-日志联动区 → 关联分析区 |
| 首期重点场景 | 登录异常 |
| 首期范围 | `P0 + P1 + 关联图谱` |

## 3. 视觉目标

| 目标 | 说明 |
| --- | --- |
| 专业感 | 像成熟后台风控产品，而不是营销页或通用详情页 |
| 判断导向 | 视觉重心要服务“快速核实和判断”，不是堆信息 |
| 信息层级清晰 | 用户能快速区分总览、场景切换、关键指标、依据、时间线、日志、图谱 |
| 可读性强 | 高密度信息下仍然易扫读、易定位 |
| 兼容现有后台风格 | 不引入过于跳脱的新组件语言，优先稳健、清晰、专业 |

## 4. 出图重点约束

| 项目 | 约束 |
| --- | --- |
| 不改结构 | 不新增/删除已确认主模块 |
| 不做全屏 | 必须是大弹窗，不是全屏工作台 |
| 不加二次浮层 | 设计中不要依赖弹窗内再开抽屉/小弹窗 |
| 不把基础信息做重 | 基础信息区域必须是一行三列收起态，不可膨胀成资料大卡 |
| 不把概览卡做成资料卡 | 登录异常概览卡必须是纯数值指标卡 |
| 不让图谱抢主视区 | 图谱是深挖模块，不是首屏第一视觉焦点 |

## 5. 首轮出图方向

首轮建议至少出 3 个方向，重点比较信息密度和视觉层级，而不是改结构。

| 方向 | 主题 | 重点 |
| --- | --- | --- |
| A | 稳健后台型 | 最像成熟风控平台，层级清楚，风险色使用克制 |
| B | 数据分析型 | 更强调时间线、图谱、指标卡的数据感 |
| C | 核实工作台型 | 更强调“当前正在核实”的流程感和操作指引 |

当前已选设计倾向：

| 项目 | 结论 |
| --- | --- |
| 首选方向 | A：稳健后台型 |
| 风格约束 | 需与现有查询页面风格统一，不能像另一套产品 |

## 5.1 参考图风格提取

基于当前已提供的原始页面图，风格特征提取如下：

| 维度 | 风格特征 |
| --- | --- |
| 产品气质 | 成熟中文后台系统，稳健、可信、偏风控业务 |
| 顶部导航 | 深蓝色通栏导航，品牌感较强，但整体克制 |
| 页面底色 | 大面积白底 + 很浅的冷灰背景，通透、干净 |
| 容器样式 | 大量圆角白色卡片，边界轻、阴影浅、层次柔和 |
| 主色 | 高饱和蓝色用于主按钮、选中Tab、强调交互 |
| 风险色 | 红/橙只用于风险提醒和高危标签，面积小但醒目 |
| 信息密度 | 中高密度，但通过留白、分组、边框和表格行高保持可读 |
| 文本层级 | 标题偏深、偏粗；说明文字偏浅灰蓝；数字和状态更突出 |
| 表格风格 | 轻边框、浅底、行高舒适，标签和图标辅助识别 |
| 图标语言 | 线性或简洁面性图标，小尺寸点缀，不花哨 |
| 交互风格 | 主操作高亮明确，其余操作克制，不做夸张悬浮效果 |

## 5.2 与参考页保持统一的视觉要求

| 项目 | 要求 |
| --- | --- |
| 弹窗风格 | 必须像从当前页面里点击查询后自然打开的结果弹窗，而不是另一套独立系统 |
| 圆角与阴影 | 延续参考图中的大圆角、浅阴影、轻边框卡片语言 |
| 蓝色主操作 | 延续参考图的主按钮蓝、选中态蓝、导航蓝 |
| 风险表达 | 延续小面积红/橙风险提示，不要整页大面积铺风险色 |
| 字体层级 | 保持中文后台风格，标题、说明、数据、标签层级清楚 |
| 密度控制 | 延续“信息较多但不压迫”的气质，避免做得过满或过空 |

## 5.3 出图方式建议

针对“页面内点击查询后打开弹窗”的关系，建议优先使用以下两种方式之一：

| 方式 | 建议程度 | 说明 |
| --- | --- | --- |
| 方式 A：一次生成同页联动图 | 推荐 | 同一张图里展示原页面背景 + 打开的查询结果弹窗，最容易保证风格统一 |
| 方式 B：一次生成两张配套图 | 可选 | 一张是查询页，一张是打开弹窗后的状态，适合分别细看，但要更明确要求风格继承 |

当前更推荐：`方式 A`

原因：

1. 更容易保证“这是同一系统里的弹窗”
2. 更利于直接评估弹窗与原页面的关系
3. 首轮出图目标是看整体风格是否统一，而不是拆开逐页精修

## 6. 通用主提示词

```text
设计一个中文后台产品的高保真界面，主题为“多维画像查询”。请采用成熟风控后台产品风格，延续现有查询页面的视觉语言：深蓝色顶栏、白色大卡片、浅阴影、较大圆角、主操作高亮蓝色、风险提示小面积使用红橙色、整体通透干净但信息密度较高。

请优先生成“页面背景 + 打开的查询结果大弹窗”的同页联动效果图，也就是用户在查询页面点击查询后打开弹窗的状态。背景页面与弹窗必须明显属于同一套系统，风格不能割裂。

查询结果容器必须是大弹窗，不是全屏页面。整体风格应专业、克制、清晰，符合成熟后台系统规范，强调信息层级和可读性，不做营销风、炫技风或夸张插画。

页面从上到下的固定结构如下：
1. 弹窗标题区：标题带查询对象，右上角关闭按钮
2. 基础信息区域：默认收起，一行三列概览；查询账号时展示 SDKID、账号名、账号状态
3. 场景总览条：展示核实状态、各场景异常数、推荐场景、推荐依据一句话
4. 场景 Tab：固定顺序为“登录异常 / 注册异常 / 支付异常”，推荐场景带“推荐查看”标记
5. 登录异常概览卡：纯数值指标卡，只展示“近7天风险登录次数 / 近7天最高风险分 / 近7日平均风险分”
6. 核实依据区：展示3到5条结构化短句依据，解释为什么这样判断
7. 时间线-日志联动区：上方是时间轴，下方是日志表
   - 时间线横轴为时间，纵轴为风险分
   - 只对风险节点做特殊标记
   - 风险节点点击后在节点下方内联展开详情
   - 展开字段包括：事件时间、风险分、登录账号、设备信息、IP信息、具体得分项
   - 日志区默认展示近7天风险记录，可切换全部记录，风险记录高亮
8. 关联分析区：上方显示关联统计副标题，下方是关联图谱
   - 查询账号时，副标题文案为：“账号 XXXXXXX 关联了 X 个设备，关联了 X 个 IP”
   - 图谱节点移入显示：对象唯一标识、近7天风险登录次数、近7天最高风险分、近7日平均风险分
   - 图谱节点点击可用于下钻分析

请特别注意：
- 页面目标是帮助用户快速形成判断，而不是堆砌资料
- 页面与弹窗的圆角、阴影、按钮、Tab、表格风格必须明显继承自同一套后台设计语言
- 基础信息区域要轻，不要做成大资料卡
- 场景概览卡是数值指标卡，不要重复展示“是否存在异常”的布尔结论
- 核实依据区要比概览卡更轻解释、比日志区更强概括
- 时间线要明显体现风险节点，但不要做成复杂分析器
- 图谱是深挖模块，不能抢走场景总览和概览卡的视觉优先级
- 所有文案、标签、表格、标题均为中文
- 设计稿要体现高信息密度下的清晰层级、间距、状态、对齐和专业感
```

## 7. 三个方向的补充提示词

### 7.1 方向 A：稳健后台型

```text
在通用主提示词基础上，采用成熟后台产品风格，并强烈参考现有查询页面的视觉气质：深蓝导航、白底大卡片、蓝色主按钮、浅阴影、温和圆角、轻量边框。重点突出秩序感、间距、对齐、清晰边界和表格可读性。风险色克制使用，只在关键风险节点、风险数字和风险标签上强调。整体以稳健、可信、长期可用为优先。
```

## 7.4 如果要一次生成“两张配套图”的补充提示词

```text
请生成两张同一设计体系下的配套高保真界面：
第一张：多维画像查询主页，展示查询入口、历史查询数据、近期高危查询等内容；
第二张：用户在第一页点击查询后打开的大弹窗结果页。

要求两张图必须明显属于同一套后台系统：
- 顶栏、侧边栏、卡片、按钮、Tab、表格、阴影、圆角、配色都保持一致
- 第二张图的弹窗应像从第一页自然弹出，而不是另一个产品
- 风格优先采用稳健后台型
```

### 7.2 方向 B：数据分析型

```text
在通用主提示词基础上，增强数据分析感。让指标卡、时间线和图谱更有数据产品气质，但仍保持后台专业感。重点比较数值、趋势、风险节点和日志信息的可扫读性，避免花哨可视化。
```

### 7.3 方向 C：核实工作台型

```text
在通用主提示词基础上，更强调“当前正在核实”的工作流体验。场景总览条、推荐查看标记、核实依据区应更有导向感，让运营一眼知道先看哪里、看完能做什么。仍需保持专业、克制，不要变成运营面板或客服面板。
```

## 8. 首轮评审维度

出图后建议按以下维度比较：

| 维度 | 检查点 |
| --- | --- |
| 结构清晰度 | 是否一眼看清从总览到场景到明细的顺序 |
| 首屏判断力 | 是否能快速看到异常总况和场景重点 |
| 概览卡表达 | 是否保持“纯数值指标卡”，没有变成详情卡 |
| 核实依据区层级 | 是否既可见又不抢概览卡主导权 |
| 时间线可读性 | 风险节点是否清楚，展开后是否不显乱 |
| 日志区可用性 | 风险记录、全量切换、筛选和高亮是否自然 |
| 图谱优先级 | 图谱是否后置且有价值，而不是喧宾夺主 |
| 后台专业感 | 是否符合成熟中文后台产品风格 |

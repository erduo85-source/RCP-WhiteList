# 多维画像查询可执行方案

## 1. 判断

| 项目 | 结论 |
| --- | --- |
| 计划方向 | 合理，可以作为当前永久分支树下的主线任务推进 |
| 主要问题 | 原计划缺少“需求收敛输出物”“页面骨架验收口”“开发前规格定稿”三层收口 |
| 推荐策略 | 先收需求和结构，再做视觉，再做实现，最后多轮验收后提交 |
| 实施原则 | 先保证可看懂、可点击、可评审，再补齐视觉精细度和代码整理 |

## 2. 目标

在现有“画像中心”下新增“多维画像查询”二级入口，完成从需求对齐、页面方案、视觉定稿到前端实现和验收提交的完整闭环。

| --- | --- |
| 导航入口 | 现有 `src/App.tsx` 中已存在“画像中心”一级菜单 |

## 3.4 当前首屏压缩结论

| 项目 | 说明 |
| --- | --- |
| 已识别问题 | 默认高度下首屏信息过深，`关联分析` 与 `登录日志` 可见性不足 |
| 本轮结构决策 | 移除 `轻量概览条 + 指标卡`，改为在 `登录分布` 标题下放一行摘要 |
| 新结构基线 | 标题栏 → 基础信息收起栏 → 场景页签 → 登录分布（含摘要） → 关联分析 → 登录日志 |

| 后续影响 | 视觉迭代和实现阶段均以“主链路优先”的精简结构为基线，不再回到摘要优先结构 |

## 5. 基础需求问询清单

建议按下表完成多轮问询，问完后再进入页面方案。

| 主题 | 必问问题 | 影响 |
| --- | --- | --- |
| 页面目标 | 这个页面主要解决什么问题？ | 决定页面主线 |
| 使用角色 | 主要由产品、运营还是风控使用？ | 决定信息密度和术语表达 |
| 查询对象 | 查单个用户、用户群，还是某类行为样本？ | 决定查询入口和结果布局 |
| 查询维度 | 计划支持哪些维度？标签、行为、设备、地域、风险等级、时间等？ | 决定筛选区规模 |
| 结果用途 | 结果用于看详情、筛人群、导出、对比，还是触发后续动作？ | 决定结果区形态 |
| 历史能力 | 是否需要保存查询、常用组合、历史记录？ | 决定交互复杂度 |
| 边界状态 | 无结果、无权限、数据延迟、加载失败如何处理？ | 决定状态设计 |
| 后续动作 | 是否需要导出、加入名单、生成分群、复制条件？ | 决定操作栏设计 |

## 6. 页面方案建议

### 6.1 页面结构

| 模块 | 建议内容 |
| --- | --- |
| 顶部区 | 页面标题、功能说明、最近更新时间、查询历史入口 |
| 查询区 | 关键词检索、维度筛选、时间范围、标签组合、快捷筛选 |
| 条件管理区 | 已选条件展示、清空、保存查询、恢复最近一次查询 |
| 结果区 | 用户列表/人群列表、标签摘要、核心风险特征、关键指标 |
| 详情补充区 | 单个结果展开后的基础属性、行为概览、画像标签、风险解释 |
| 辅助区 | 查询说明、数据口径、示例查询 |
| 状态区 | 默认态、加载态、空态、异常态 |

### 6.2 交互优先级

| 优先级 | 建议内容 |
| --- | --- |
| P0 | 能完成一次完整查询并看到结果 |
| P0 | 条件变更后有明确反馈，支持清空和重置 |
| P0 | 结果为空、加载中、失败时有可理解反馈 |
| P1 | 支持保存常用查询 / 最近查询 |
| P1 | 支持查看单个结果的更多画像信息 |
| P2 | 导出、复制查询条件、分群对比等增强能力 |

## 7. 视觉设计阶段建议

| 项目 | 建议 |
| --- | --- |
| 出图时机 | 仅在页面骨架和交互规则确认后开始 |
| 出图方式 | 每轮至少出 2 个方向，不要只出 1 个方案 |
| 对比维度 | 信息密度、专业感、可读性、条件区效率、结果区层次 |
| 重点检查 | 表单对齐、标签密度、卡片层次、表格可读性、状态页完整性 |
| 定稿标准 | 用户能明确指出“用哪版继续开发”，且不再修改页面骨架 |

## 11. 风险与控制建议

| 风险 | 表现 | 控制建议 |
| --- | --- | --- |
| 需求边界不清 | 页面持续加条件、加模块 | 先锁定首版范围，增强项后置 |
| 视觉先行过早 | 设计图反复推翻结构 | 先定骨架，再出图 |
| 单文件继续膨胀 | `App.tsx` 难维护 | 开发阶段至少拆出 `features/profile-query` |
| mock 掩盖真实问题 | 展示看起来完整，但状态不真实 | 明确哪些是 mock，保留异常态和空态 |
| 验收标准不一致 | 每轮意见方向不同 | 在开发前先对齐验收口 |

## 12. 当前开发前结论

当前最合理的下一步已经不是继续问询，而是进入“任务 4：完成前端实现”。  
开发阶段应默认需求和结构已锁定；视觉若继续迭代，应以“首屏精简版结构”为当前基线，不再回退到旧的摘要优先结构。

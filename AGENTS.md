# AGENTS.md

## 使用场景

本项目主要由产品经理在本地使用，用于验证figma与AI生成的工作流
包括读取（设计规范、设计稿、组件库）、转化（设计稿转为本地可运行的html文件）、设计（在Figma内生成符合设计规范的设计稿）

## 核心原则
   
1. 默认使用中文。
2. 先给结论、方案或结果，再补充必要解释。
3. 少复述背景，少空话，避免把简单问题复杂化。
4. 如果存在不确定性，先澄清再动手。
5. 不假装完成。没有运行/验证/截图，就明确说明“未运行”或“未验证”。

## 开发工作流

1. 先判断任务级别  
   - 小改动：文案、样式、布局、交互细节、配置修正，可直接执行。  
   - 中等改动：少量多文件、低风险逻辑调整，先简短列计划再执行。  
   - 大改动：涉及多页面、多流程、目录结构、核心状态、依赖安装、外部接口或数据结构，先规划再执行。
2. 动手前尽量拆成四段  
   `files`：影响哪些文件；`action`：做什么；`verify`：如何验证；`done`：完成后应看到什么。
3. 高风险操作先确认  
   删除文件、批量覆盖、安装依赖、改系统配置、运行不可逆命令、创建 branch / worktree 前，必须先说明风险并等确认。
4. 改完必须验证  
   优先本地运行、构建、类型检查或手动交互验证。若无法验证，说明原因、剩余风险和建议验证方式。

## Skill工具使用

1. 以当前会话或当前环境实际暴露的 skill、tool、agent 为准。不要假装使用了未暴露或不可用的能力。
2. 如果可用工具能提升任务质量，可以优先使用；如果不可用，按本文档的工作流手工推进。
3. 复杂任务可用 `<files>/<action>/<verify>/<done>` 维护简短计划和进度。
4. 实际使用了某个 skill / tool时，在回复中说明使用情况和原因。






## 输出规范

1. 优先表格化  
   只要涉及判断、结论、对比、步骤、风险、文件变更、验证结果，优先用 Markdown 表格呈现，让用户快速扫读。
2. 简单问题短答；复杂问题按“判断 -> 方案 -> 步骤 -> 风险/验证”组织。
3. 用户要方案时，给清晰结构和可复制内容；用户要实现时，说明改了什么、怎么运行、如何验证。
4. 不捏造测试、运行结果或数据来源。没有实际运行就说“未运行”。
